Locking and deadlocks

MVCC ขจัดความจำเป็นในการ lock เพื่ออ่าน แต่การเขียนยังต้องถูกประสานงานอยู่ดี เมื่อสอง transaction ต้องการเปลี่ยนแถวเดียวกัน ตัวหนึ่งต้องรอให้อีกตัวทำเสร็จ ส่วนใหญ่แล้ว PostgreSQL จับ row lock ที่จำเป็นให้คุณในระหว่าง UPDATE หรือ DELETE แต่บางครั้งคุณก็อยาก lock แถวในขณะที่อ่านมัน เพื่อให้ไม่มีใครเปลี่ยนมันได้ก่อนที่คุณจะเปลี่ยน

จองแถวด้วย FOR UPDATE

หัวข้อที่มีชื่อว่า “จองแถวด้วย FOR UPDATE”

SELECT ... FOR UPDATE อ่านแถวและ lock มันราวกับว่าคุณกำลังจะ update มัน transaction อื่นใดที่พยายาม update, delete หรือ lock แถวนั้นด้วยจะต้องรอจนกว่าคุณจะ commit หรือ roll back นี่คือรูปแบบมาตรฐานสำหรับ “อ่านยอดเงิน ตัดสินใจ แล้วเขียนกลับ” โดยไม่เกิด lost update

  • Node.js
  • Python
  • Go
  • Rust

lock คู่หูคือ FOR SHARE มันยอมให้คนอื่นอ่านและจับ FOR SHARE ได้ด้วย แต่ block ใครก็ตามที่ต้องการ FOR UPDATE หรือการเขียน ใช้มันเมื่อคุณต้องรับประกันว่าแถวจะไม่เปลี่ยนใต้มือคุณ แต่คุณไม่ได้ตั้งใจจะเขียนมันเอง

lock wait

หัวข้อที่มีชื่อว่า “lock wait”

เมื่อ transaction ขอ lock ที่อีกตัวถืออยู่ มันก็แค่รอ query ที่กำลังรอจะดูเหมือนค้าง มันจะกลับมาทำงานทันทีที่ผู้ถือ commit หรือ roll back การถือสั้น ๆ มองไม่เห็นจากผู้ใช้ ส่วนการถือนาน ๆ กลายเป็นคิว

เมื่อการรอกลายเป็น deadlock

หัวข้อที่มีชื่อว่า “เมื่อการรอกลายเป็น deadlock”

deadlock เกิดขึ้นเมื่อสอง transaction ต่างก็ถือ lock ที่อีกตัวต้องการ Transaction A lock แถว 1 แล้วต้องการแถว 2 ส่วน transaction B lock แถว 2 แล้วตอนนี้ต้องการแถว 1 ไม่มีตัวใดเดินหน้าต่อได้ และไม่มีตัวใดยอมปล่อยก่อน

PostgreSQL รัน deadlock detector อยู่ หลังจาก timeout สั้น ๆ มันจะสังเกตเห็นวงจร เลือก transaction หนึ่งเป็นเหยื่อ และ abort มันด้วย error เพื่อให้อีกตัวทำงานต่อได้ เหยื่อจะเห็น SQLSTATE 40P01 และควร retry ทั้ง transaction ของมัน

เลือกที่จะไม่รอ: NOWAIT และ SKIP LOCKED

หัวข้อที่มีชื่อว่า “เลือกที่จะไม่รอ: NOWAIT และ SKIP LOCKED”

บางครั้งการรอเป็นพฤติกรรมที่ผิด ตัวขยายสองตัวเปลี่ยนมันได้ NOWAIT ทำให้คำสั่ง fail ทันทีแทนที่จะเข้าคิวหากแถวถูก lock อยู่ ส่วน SKIP LOCKED ก้าวข้ามแถวที่ถูก lock อย่างเงียบ ๆ และคืนเฉพาะแถวที่มันสามารถ lock ได้ ซึ่งเป็นแกนหลักของรูปแบบ work-queue ที่ worker หลายตัวต่างก็คว้างานว่างตัวถัดไป

  • psql
  • Node.js
  • Python
  • Go
  • Rust

ใน pgAdmin: เพื่อสังเกต lock wait สด ๆ ให้รัน BEGIN; SELECT ... FOR UPDATE ในแท็บ Query Tool หนึ่งโดยไม่ commit แล้วรันอันเดียวกันในแท็บที่สองและดูมันค้าง view pg_locks แสดงว่าใครถืออะไรอยู่ในขณะที่คุณทดลอง

Tips / gotchas

หัวข้อที่มีชื่อว่า “Tips / gotchas”
  • lock แถวในลำดับที่สอดคล้องกันทุกที่ในโค้ดของคุณ ถ้าทุก transaction แตะแถว 1 ก่อนแถว 2 วงจรที่ก่อให้เกิด deadlock ก็จะไม่มีวันก่อตัวขึ้นได้
  • ปฏิบัติต่อ deadlock error (40P01) และ serialization failure (40001) แบบเดียวกัน คือ roll back แล้ว retry ทั้ง transaction ทั้งสองเป็นเรื่องที่คาดหมายได้ภายใต้ concurrency
  • ทำให้งานระหว่างการจับ lock กับการ commit สั้นที่สุดเท่าที่จะทำได้ ยิ่งคุณถือ lock นานเท่าใด คนอื่นก็เข้าคิวต่อท้ายคุณนานขึ้นเท่านั้น
  • SKIP LOCKED เหมาะอย่างยิ่งสำหรับคิว แต่ผิดสำหรับงานบัญชี มันจงใจเพิกเฉยต่อแถว ดังนั้นอย่าใช้มันในที่ที่คุณต้องเห็นทุกแถวที่ตรงกัน

🐘 ลำดับ lock ที่สอดคล้องกันป้องกัน deadlock ส่วนใหญ่ได้

deadlock ไม่ได้เกิดจากการ lock มากเกินไป แต่เกิดจากการ lock ในลำดับที่ขัดแย้งกัน จงตัดสินใจลำดับสากลหนึ่งลำดับสำหรับการจับ lock เช่นเรียงตาม primary key จากน้อยไปมากเสมอ และนำไปใช้ในทุก transaction PostgreSQL จะยังตรวจจับ deadlock ใด ๆ ที่หลุดรอดมาได้ แต่คุณจะแทบไม่ส่งมันให้ไปค้นเจอเลย
