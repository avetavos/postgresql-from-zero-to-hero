เก็บ JSON ด้วย JSONB

บางครั้งแถวหนึ่งต้องเก็บข้อมูลที่รูปร่างไม่ได้ถูกกำหนดไว้ล่วงหน้า เช่น สินค้าที่มีคุณสมบัติต่างกันไปตามหมวดหมู่ payload ของ event จากบุคคลที่สาม หรือการตั้งค่าของผู้ใช้ที่งอกเพิ่มขึ้นตามเวลา PostgreSQL ให้คอลัมน์เดียวเก็บเอกสาร JSON ทั้งก้อนด้วยชนิดข้อมูล jsonb แล้ว query เข้าไปข้างในได้ราวกับว่า key ของมันคือคอลัมน์

ในบทนี้เราใช้ตารางของ products ที่แต่ละแถวพกเอกสาร attributes ที่ยืดหยุ่นไว้

การสร้างคอลัมน์ JSONB

หัวข้อที่มีชื่อว่า “การสร้างคอลัมน์ JSONB”

ชนิดข้อมูล jsonb เก็บ JSON ในรูปแบบไบนารีที่ถูกแยกย่อยแล้ว (decomposed binary form) เมื่อเทียบกับชนิด json ธรรมดา มันจะตัด whitespace ที่ไม่สำคัญทิ้ง ลบ key ซ้ำ ไม่รักษาลำดับของ key และ — ที่สำคัญที่สุด — สามารถทำ index ได้

การเก็บเอกสาร

หัวข้อที่มีชื่อว่า “การเก็บเอกสาร”

การ insert แถวก็เป็น INSERT ธรรมดา เอกสารเป็นเพียง JSON string ที่ PostgreSQL parse ให้เป็น jsonb จากฝั่ง driver คุณส่ง JSON เป็นพารามิเตอร์แทนการต่อ string ด้วยมือ

  • psql
  • Node.js
  • Python
  • Go
  • Rust

ใน pgAdmin: เซลล์ attributes จะแสดงเอกสาร JSON ดับเบิลคลิกที่มันเพื่อเปิดมุมมอง editor ที่ใหญ่ขึ้นสำหรับดูค่า

การเข้าถึงข้างในเอกสาร

หัวข้อที่มีชื่อว่า “การเข้าถึงข้างในเอกสาร”

operator ไม่กี่ตัวช่วยดึงค่าออกมาจากค่า jsonb สองตัวที่คุณจะหยิบใช้บ่อยที่สุดคือ -> และ ->>

Operatorคืนค่าตัวอย่าง
->ค่า jsonb (object หรือ array)attributes -> 'sizes'
->>ค่าในรูปแบบ textattributes ->> 'color'
#>ค่า jsonb ที่ path หนึ่งattributes #> '{sizes,0}'
#>>text ที่ path หนึ่งattributes #>> '{sizes,0}'

ความต่างระหว่าง -> กับ ->> นั้นสำคัญ คือ -> คงผลลัพธ์ไว้เป็น jsonb (ดีสำหรับการต่อลงไปในข้อมูลที่ซ้อนลึกขึ้น) ส่วน ->> ให้ text ธรรมดาที่คุณนำไปเทียบกับ string หรือ cast เป็นตัวเลขได้

  • psql
  • Node.js
  • Python
  • Go
  • Rust

การกรองด้วย containment และการทดสอบ key

หัวข้อที่มีชื่อว่า “การกรองด้วย containment และการทดสอบ key”

operator สองตัวถูกสร้างมาเพื่อการค้นหา containment operator @> ถามว่าเอกสารฝั่งซ้ายมีเอกสารฝั่งขวาอยู่หรือไม่ ส่วน existence operator ? ถามว่า key ระดับบนสุดมีอยู่หรือไม่

  • psql
  • Node.js
  • Python
  • Go
  • Rust

การแก้ไขบางส่วนของเอกสาร

หัวข้อที่มีชื่อว่า “การแก้ไขบางส่วนของเอกสาร”

หากต้องการเปลี่ยนฟิลด์เดียวโดยไม่ต้องเขียนเอกสารทั้งก้อนใหม่ ให้ใช้ jsonb_set มันรับคอลัมน์ path ในรูปแบบ text array และค่าใหม่ในรูปแบบ jsonb

หากต้องการเพิ่มฟิลด์ใหม่เอี่ยม คุณยังใช้ concatenation operator ได้ด้วย: attributes || '{"on_sale": true}' จะรวมเอกสารฝั่งขวาเข้าไปในฝั่งซ้าย

การทำ index ให้ containment query

หัวข้อที่มีชื่อว่า “การทำ index ให้ containment query”

หากไม่มี index คำสั่ง @> จะ scan ทุกแถวและทดสอบแต่ละเอกสาร GIN index เปลี่ยนเรื่องนั้น มันทำ index ให้กับ key และ value ภายใน jsonb ดังนั้นการค้นหาแบบ containment และ key-existence จึงเร็วขึ้น

หลังจากนี้ query ที่ใช้ @>, ?, ?| และ ?& จะใช้ index ได้ ถ้าคุณ query เฉพาะ path ใดเพียง path เดียวเสมอ expression index บน path นั้นจะเล็กกว่าและเร็วกว่าการทำ index ให้ทั้งเอกสาร

Tips / ข้อควรระวัง

หัวข้อที่มีชื่อว่า “Tips / ข้อควรระวัง”
  • เลือก jsonb แทน json สำหรับเกือบทุกกรณี ชนิด json เก็บข้อความเป๊ะ ๆ ตามที่คุณส่งมา (รักษา whitespace และลำดับ) แต่ทำ index ไม่ได้ และ parse ใหม่ทุกครั้งที่เข้าถึง
  • ใช้ jsonb กับข้อมูลที่แปรผันจริง ๆ ไม่ใช่เป็นทางเลี่ยงการออกแบบ schema ถ้าฟิลด์หนึ่งมีอยู่เสมอและเป็นชนิดเดิมเสมอ คอลัมน์จริงจะให้คุณได้ทั้งการตรวจสอบชนิด ค่า default และ constraint
  • เพิ่ม GIN index เฉพาะเมื่อคุณ query เข้าไปข้างในเอกสารจริง ๆ การทำ index ให้ข้อมูลที่คุณอ่านกลับมาทั้งก้อนเท่านั้นเป็นการเปลืองพื้นที่และเพิ่มภาระตอนเขียน
  • จำไว้ว่า -> คืน jsonb และ ->> คืน text การเทียบ attributes -> 'color' = 'teal' จะล้มเหลวเพราะฝั่งซ้ายเป็น jsonb จงใช้ ->> สำหรับการเทียบ text

🐘 Document store ภายในฐานข้อมูลเชิงสัมพันธ์ของคุณ

คอลัมน์ jsonb ทำให้ตารางเดียวผสมคอลัมน์ที่เคร่งครัดและมีชนิดข้อมูลชัดเจน เข้ากับฟิลด์เอกสารที่ยืดหยุ่นได้ ความเป็นลูกผสมนั้นมักเป็นความยืดหยุ่นของ schema ทั้งหมดที่แอปพลิเคชันต้องการ — ไม่ต้องมี document database แยกต่างหาก เพียงแค่เก็บส่วนที่คงที่จริง ๆ ไว้ในคอลัมน์จริงที่ Postgres บังคับใช้ได้
