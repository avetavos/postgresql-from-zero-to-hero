การเรียก `find` ตอบคำถามได้แบบเดียว: "คืนเอกสารที่ตรงกับฟิลเตอร์นี้กลับมาให้ฉัน" ซึ่งเพียงพอสำหรับการอ่านข้อมูล แต่มันไม่สามารถ *สรุป* ข้อมูลเหล่านั้นได้ มันบอกคุณไม่ได้ว่าค่าเฉลี่ย rating ต่อ genre เป็นเท่าไร ลูกค้าแต่ละคนสั่งซื้อกี่ครั้ง หรือหนังสือสามอันดับขายดีที่สุดเมื่อเดือนที่แล้วคืออะไร สำหรับคำถามที่ต้องยุบเอกสารจำนวนมากให้กลายเป็นคำตอบที่เล็กลงและถูกจัดรูปใหม่ MongoDB มอบ **aggregation pipeline** ให้คุณ

pipeline เป็นแนวคิดที่ทรงพลังที่สุดใน MongoDB รองจากตัวเอกสารเอง ลองนึกภาพสายพานลำเลียง: เอกสารของคุณเข้ามาทางปลายด้านหนึ่ง ไหลผ่านชุดของ **stage** แล้วสตรีมที่ถูกแปลงแล้วออกมาทางปลายอีกด้าน แต่ละ stage รับเอกสารที่ส่งมาให้ ทำงานหนึ่งอย่างที่นิยามไว้ชัดเจน แล้วส่งผลลัพธ์ต่อไปยัง stage ถัดไป ลำดับเป็นสิ่งที่คุณเลือกได้ และลำดับนั้นสำคัญ — stage หนึ่งจะเห็นเฉพาะสิ่งที่ stage ก่อนหน้ามันผลิตออกมาเท่านั้น

เราจะทำงานกับชุดข้อมูลเล็ก ๆ ที่สมมติขึ้นตลอดทั้งโมดูลนี้: คอลเลกชัน `sales` ของออเดอร์หนังสือในร้านในจินตนาการ การทำให้ข้อมูลมีขนาดเล็กช่วยให้ *รูปร่าง* ของการแปลงแต่ละครั้งโดดเด่นชัดเจน

## pipeline คือลิสต์ของ stage

คุณส่ง **array** ของออบเจกต์ stage ให้กับ aggregation แต่ละ stage ถูกตั้งชื่อด้วยคีย์ที่นำหน้าด้วย `$` — `$match`, `$group`, `$sort` เป็นต้น นี่คือรูปแบบมาตรฐาน: ฟิลเตอร์สตรีม ยุบมันเข้ากลุ่ม แล้วเรียงลำดับผลลัพธ์

array ของ stage ชุดเดียวกันนี้คือสิ่งที่คุณส่งเข้าไปในทุกภาษา สังเกตว่าการเรียกคือ `aggregate` และอาร์กิวเมนต์คือ pipeline:

    pipeline := mongo.Pipeline{
        {{"$match", bson.D{{"genre", "fiction"}}}},
        {{"$group", bson.D{{"_id", "$title"}, {"copies", bson.D{{"$sum", "$quantity"}}}}}},

    cursor, err := coll.Aggregate(ctx, pipeline)

    }

**ใน Compass:** เปิดคอลเลกชันแล้วคลิกแท็บ **Aggregations** ตัวสร้าง pipeline ให้คุณเพิ่มทีละ stage จาก dropdown แก้ไขเนื้อหาของมัน และดูตัวอย่างเอกสารที่ไหลออกจากแต่ละ stage แบบสดในตัวอย่างข้อมูล — นี่เป็นวิธีที่ดีที่สุดในการ *เฝ้าดู* สายพานลำเลียงทำงาน

## ต่างจาก find อย่างไร

`find` ฟิลเตอร์และ project แต่มันจะคืนสับเซตของเอกสาร *ต้นฉบับ* ของคุณเสมอ aggregation ทำแบบนั้นได้เช่นกัน แต่มันยังสามารถสร้างเอกสารใหม่ทั้งหมดขึ้นมาได้ด้วย: จำนวนนับ ค่าเฉลี่ย ระเบียนที่ join แล้ว อาร์เรย์ที่แบนราบ อะไรก็ตามที่ `find` ทำได้ aggregation ก็ทำได้ด้วย `$match` และ `$project` แต่ในทางกลับกันไม่จริง

- `find` อ่านเอกสารออกมา; aggregation **จัดรูปสตรีม** ของเอกสารใหม่
- `find` คืนเอกสารที่มีอยู่แล้ว; aggregation สามารถ **สังเคราะห์** เอกสารใหม่ขึ้นมาได้
- pipeline เป็นสิ่งที่ **เรียงลำดับและประกอบกันได้** — คุณเรียงซ้อน stage เหมือนต่อ Lego และเพิ่มได้เรื่อย ๆ

## โมดูลนี้ครอบคลุมอะไรบ้าง

บทเรียนข้างหน้าจะแนะนำ stage ที่คุณจะหยิบมาใช้บ่อยที่สุด ทีละแนวคิด:

1. **Pipeline basics** — `$match` เพื่อฟิลเตอร์ตั้งแต่เนิ่น ๆ และ `$project` เพื่อจัดรูปใหม่ และทำไมลำดับของ stage จึงเปลี่ยนทั้งความถูกต้องและความเร็ว
2. **Group and accumulators** — `$group` พร้อม `$sum`, `$avg`, `$min`, `$max`, `$push` และพวกพ้อง บวกกับยอดรวมทั้งหมด
3. **Lookup joins** — `$lookup` เพื่อดึงเอกสารที่เกี่ยวข้องจากคอลเลกชันอื่น
4. **Unwind arrays** — `$unwind` เพื่อเปลี่ยนหนึ่งเอกสารที่มีอาร์เรย์ให้กลายเป็นหลายเอกสาร
5. **Facets and buckets** — `$facet` สำหรับหลายคำตอบในการประมวลผลครั้งเดียว และ `$bucket` สำหรับการจัดกลุ่มแบบฮิสโทแกรม
