ทุกตารางเริ่มต้นชีวิตด้วยคำสั่ง `CREATE TABLE` คุณตั้งชื่อตาราง จากนั้นจึงไล่รายชื่อ column ของมันภายในวงเล็บ โดยให้แต่ละตัวมีชื่อและ data type เซิร์ฟเวอร์จะเก็บคำนิยามนั้นไว้ และนับจากนั้นก็จะบังคับใช้กับทุกแถวที่คุณ insert การสร้างตารางคือ **DDL** — data definition language — และมักจะเขียนและรันเป็น SQL ธรรมดา ไม่ว่าคุณจะพิมพ์ลงใน `psql` หรือป้อนให้กับเครื่องมือ migration

เราจะสร้างตาราง `products` สำหรับร้านค้าเล็ก ๆ และทำให้มันเติบโตขึ้นไปเรื่อย ๆ ตามที่บทเรียนดำเนินไป

## รูปทรงพื้นฐาน

อย่างน้อยที่สุดคือชื่อตารางและ column หนึ่งตัวขึ้นไป แต่ละ column คือชื่อตามด้วยชนิด โดย column ต่าง ๆ คั่นด้วยจุลภาค

```sql
CREATE TABLE products (
  id     integer,
  name   text,
  price  numeric(10, 2)
);
```

นั่นสร้างตารางที่มีสาม column: `id` ที่เป็นจำนวนเต็ม, `name` ที่เป็น text และ `price` ที่เก็บเป็นทศนิยมแบบแม่นยำได้ถึงสิบหลักและสองหลักหลังจุด มันใช้ได้ แต่หลวมเกินไป — ไม่มีอะไรห้ามสินค้าสองชิ้นใช้ `id` ร่วมกัน และทุก column อาจถูกปล่อยว่างไว้ได้ เราจะแก้ปัญหานั้นด้วย primary key และ constraints ในลำดับถัดไป

**ใน pgAdmin:** หลังจากรันคำสั่งใน Query Tool ให้ refresh โหนด Tables ในแผนผัง browser แล้วตาราง `products` ใหม่จะปรากฏขึ้น เมื่อขยายมันออกจะเผยให้เห็นโหนด Columns ที่ไล่รายชื่อแต่ละ column และชนิดของมัน

## ตรวจสอบตารางด้วย backslash-d

ใน `psql` คำสั่ง `\d` จะอธิบายตาราง: column ของมัน, ชนิดของมัน, ความสามารถในการเป็น null และ default ใด ๆ มันเป็นวิธีที่เร็วที่สุดในการยืนยันว่าตารางมีหน้าตาอย่างที่คุณตั้งใจไว้

```sql
\d products
```

```text
                        Table "public.products"
 Column |     Type      | Collation | Nullable | Default
--------+---------------+-----------+----------+---------
 id     | integer       |           |          |
 name   | text          |           |          |
 price  | numeric(10,2) |           |          |
```

ตอนนี้ทุก column สามารถเป็น null ได้และยังไม่มี default เมื่อเราเพิ่ม primary key และ constraints เซลล์เหล่านั้นจะถูกเติมเต็มขึ้นมากขึ้น

## สร้าง primary key โดยอัตโนมัติ

คุณแทบไม่อยากกำหนดเลข id ด้วยมือเลย PostgreSQL ทำให้คุณได้ วิธีสมัยใหม่คือ `GENERATED ALWAYS AS IDENTITY` ซึ่งทำให้เซิร์ฟเวอร์จัดหาค่าถัดไปให้โดยอัตโนมัติ ทางลัดแบบเก่าคือ `SERIAL` ซึ่งคุณยังคงพบเห็นได้ในโค้ดที่มีอยู่จำนวนมาก

```sql
CREATE TABLE products (
  id     bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  name   text NOT NULL,
  price  numeric(10, 2) NOT NULL
);
```

ตอนนี้ `id` ถูกเติมโดยเซิร์ฟเวอร์ ถูกประกาศให้เป็น primary key และ `name` กับ `price` ก็ไม่สามารถถูกปล่อยว่างได้อีกต่อไป ตารางเดียวกันนี้ถ้าเขียนด้วยสไตล์เก่าจะใช้ `id bigserial PRIMARY KEY` ทั้งสองแบบให้คีย์ที่เพิ่มค่าอัตโนมัติ แบบ `IDENTITY` เป็นแบบที่เป็นมาตรฐาน SQL และเป็นที่นิยมกว่าใน PostgreSQL สมัยใหม่

## สร้างตารางจาก driver

แอปพลิเคชันส่วนใหญ่รันคำสั่ง `CREATE TABLE` เพียงครั้งเดียว ตอน setup หรือผ่านขั้นตอน migration แทนที่จะรันในทุก request นี่คือ migration เดียวกันที่ส่งจากแต่ละ driver — มีประโยชน์เมื่อแอปพลิเคชันของคุณเป็นเจ้าของ schema เอง

## IF NOT EXISTS และ DROP TABLE

การรัน `CREATE TABLE products` เป็นครั้งที่สองจะเป็น error เพราะตารางมีอยู่แล้ว การเพิ่ม `IF NOT EXISTS` ทำให้คำสั่งกลายเป็น no-op เมื่อตารางมีอยู่แล้ว ซึ่งสะดวกในสคริปต์ setup ที่อาจถูกรันมากกว่าหนึ่งครั้ง

```sql
CREATE TABLE IF NOT EXISTS products (
  id    bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  name  text NOT NULL,
  price numeric(10, 2) NOT NULL
);
```

หากต้องการลบตารางและแถวทั้งหมดของมัน ให้ใช้ `DROP TABLE` มันถาวรและเกิดขึ้นทันที จึงควรปฏิบัติกับมันด้วยความระมัดระวังเช่นเดียวกับ `DELETE` แบบ `IF EXISTS` ช่วยเลี่ยง error เมื่อตารางหายไปแล้ว

```sql
DROP TABLE IF EXISTS products;
```

**ใน pgAdmin:** คุณยัง drop ตารางได้โดยคลิกขวาที่มันในแผนผัง browser แล้วเลือก Delete/Drop แต่การรัน `DROP TABLE` ใน Query Tool จะเก็บการกระทำนั้นไว้ในประวัติสคริปต์ของคุณ

## เคล็ดลับ / ข้อควรระวัง

- เลือก `GENERATED ALWAYS AS IDENTITY` มากกว่า `SERIAL` สำหรับตารางใหม่ มันเป็นไวยากรณ์มาตรฐาน SQL หลีกเลี่ยงความแปลกประหลาดเรื่องความเป็นเจ้าของของ sequence ที่ `SERIAL` สร้างขึ้นเบื้องหลัง และอ่านได้ชัดเจนกว่า
- ใช้ `bigint` แทนที่จะเป็น `integer` สำหรับ identity key บนตารางที่คุณคาดว่าจะเติบโต `integer` ขนาดสี่ไบต์จะหมดที่ราว ๆ สองพันล้านแถว และการสลับชนิดในภายหลังนั้นสร้างความปั่นป่วน
- เพิ่ม `IF NOT EXISTS` ลงในสคริปต์สร้างที่อาจถูกรันซ้ำ ๆ แต่อย่าใส่ในการ migration แบบครั้งเดียวที่ตารางซ้ำควรเป็น error ที่คุณสังเกตเห็น
- `DROP TABLE` ลบทั้งโครงสร้างและทุกแถวโดยไม่มีการยืนยัน ตรวจสอบชื่อตารางให้ดี และควร drop ภายใน transaction ที่คุณ roll back ได้หากไม่แน่ใจ
- ชื่อ column และตารางจะถูกแปลงเป็นตัวพิมพ์เล็กเว้นแต่จะใส่ double-quote ดังนั้นจงเลือกชื่อตัวพิมพ์เล็กแบบ snake_case เพื่อหลีกเลี่ยงความประหลาดใจ

<Callout title="IDENTITY คือค่าเริ่มต้นสมัยใหม่">
เมื่อคุณมองหาคีย์ที่เพิ่มค่าอัตโนมัติ ให้เขียน `bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY` มันแสดงเจตนาได้ชัดเจน เป็นไปตามมาตรฐาน SQL และเลี่ยงพฤติกรรมแบบเก่าของ `SERIAL` ได้ เก็บ `SERIAL` ไว้สำหรับอ่านโค้ดเก่า ไม่ใช่สำหรับเขียนตารางใหม่
</Callout>
