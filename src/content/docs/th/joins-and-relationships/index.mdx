ตารางเพียงตารางเดียวตอบได้แค่คำถามเกี่ยวกับตัวมันเองเท่านั้น พลังที่แท้จริงของฐานข้อมูลเชิงสัมพันธ์จะปรากฏเมื่อคุณเลิกเก็บทุกอย่างไว้ในตารางใหญ่ตารางเดียว แล้วหันมาแบ่งข้อมูลออกเป็นตารางที่มีจุดโฟกัสชัดเจนซึ่งชี้ถึงกันและกันผ่าน key แทน ตาราง `authors` เก็บข้อมูลคน ตาราง `books` เก็บชื่อหนังสือ และ foreign key เชื่อมแต่ละหนังสือกลับไปยังผู้เขียนของมัน ไม่มีอะไรซ้ำซ้อน แต่คุณก็ยังถามคำถามที่ครอบคลุมทั้งสองตารางได้

โมดูลนี้ว่าด้วยการตอบคำถามข้ามตารางเหล่านั้น คุณจะได้เรียนรู้การเย็บตารางกลับเข้าด้วยกันด้วย join, การยุบหลายแถวให้กลายเป็นตัวเลขสรุปด้วย aggregate, การซ้อนคิวรีหนึ่งไว้ในอีกคิวรีหนึ่งด้วย subquery และ CTE, และสุดท้ายการคำนวณการจัดอันดับรายแถวและยอดสะสมด้วย window function

## สคีมาตัวอย่าง

ทุกบทเรียนจะใช้ตารางเล็ก ๆ สามตารางซ้ำกัน `authors` และ `books` น่าจะดูคุ้นเคยจากโมดูล CRUD เราเพิ่มตาราง `sales` เข้ามาเพื่อให้มีอะไรไว้ aggregate และจัดอันดับ

## ตารางสัมพันธ์กันอย่างไร

แต่ละตารางเชื่อมต่อกับตารางถัดไปผ่านคอลัมน์ key หนังสือพก `id` ของผู้เขียนเอาไว้ และยอดขายก็พก `id` ของหนังสือเอาไว้ การเชื่อมโยงเหล่านี้คือสิ่งที่ทุกคิวรีในโมดูลนี้เดินตาม

export const schemaDiagram = `flowchart LR
  A[authors] -->|id referenced by books.author_id| B[books]
  B -->|id referenced by sales.book_id| S[sales]`;

<Mermaid code={schemaDiagram} title="ตารางเชื่อมโยงกันด้วย key" />

อ่านลูกศรว่า "ถูกอ้างอิงโดย" ผู้เขียนหนึ่งคนมีหนังสือได้หลายเล่ม และหนังสือหนึ่งเล่มมีแถวยอดขายได้หลายแถว — รูปแบบ one-to-many คลาสสิกที่ join ถูกออกแบบมาเพื่อท่องผ่าน

## โมดูลนี้ครอบคลุมอะไรบ้าง

- **Joins** — รวมแถวจากสองตารางขึ้นไปบนคอลัมน์ key ที่ตรงกัน และเลือกว่าจะเก็บหรือทิ้งแถวที่ไม่มีคู่ที่ตรงกัน
- **Aggregate และ GROUP BY** — ยุบหลายแถวให้กลายเป็น count, sum, และค่าเฉลี่ย แล้วกรองกลุ่มเหล่านั้นด้วย `HAVING`
- **Subquery และ CTE** — ใช้ผลลัพธ์ของคิวรีหนึ่งภายในอีกคิวรีหนึ่ง และตั้งชื่อขั้นตอนระหว่างทางด้วย `WITH` เพื่อให้อ่านง่าย
- **Window function** — จัดอันดับแถวและคำนวณยอดสะสมโดยไม่ทิ้งแถวแต่ละแถวไป

## ทิป / จุดพลาดที่พบบ่อย

- foreign key เป็นเอกสารบันทึกความสัมพันธ์และบังคับใช้มัน แต่ join ในคิวรีของคุณต่างหากที่นำแถวที่เกี่ยวข้องมารวมกันจริง ๆ
- คำถามเดียวกันมักเขียนได้ทั้งเป็น join, subquery, หรือ window function การเรียนรู้ทั้งสามแบบทำให้คุณเลือกแบบที่ชัดเจนที่สุดได้
- ให้นึกภาพการเชื่อมโยงแบบ one-to-many ข้างต้นไว้ในใจ เพราะเกือบทุกคิวรีในโมดูลนี้เดินตามลูกศรเหล่านั้นเส้นใดเส้นหนึ่ง
