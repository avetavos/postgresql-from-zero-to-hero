เมื่อ query ทำงานช้า การเดาเป็นการเสียเวลาเปล่า PostgreSQL จะบอกคุณได้อย่างชัดเจนว่ามันตั้งใจจะรัน query อย่างไร และมันได้รันจริง ๆ อย่างไร เครื่องมือนั้นคือ `EXPLAIN` และการเรียนรู้ที่จะอ่านผลลัพธ์ของมันคือทักษะด้านประสิทธิภาพที่มีประโยชน์ที่สุดเพียงทักษะเดียวที่คุณจะหยิบจับได้

## planner เป็นผู้เลือกแผน

คุณเขียน *ว่า* คุณต้องการอะไร ส่วน **query planner** เป็นผู้ตัดสินใจ *ว่าจะ* ได้มันมาอย่างไร สำหรับ query หนึ่ง ๆ อาจมีแผนที่เป็นไปได้หลายแบบ — สแกนทั้งตาราง, ใช้ index ตัวนี้, ใช้ตัวนั้น, รวมสองตัวเข้าด้วยกัน — และ planner จะประเมินต้นทุนของแต่ละแบบ แล้วเลือกแบบที่ถูกที่สุด

ค่าประมาณต้นทุนเหล่านั้นขึ้นอยู่กับ statistics เกี่ยวกับข้อมูลของคุณ — ตารางมีกี่แถว, คอลัมน์หนึ่งมีค่าที่แตกต่างกันกี่ค่า และอื่น ๆ หาก statistics เก่าล้าสมัย planner ก็จะเดาผิดพลาด ซึ่งเป็นหัวข้อของบทเรียนสุดท้ายในโมดูลนี้

## EXPLAIN: แผนโดยประมาณ

`EXPLAIN` พิมพ์แผนที่ planner *จะ* ใช้ออกมา โดยไม่ต้องรัน query มันทำงานทันทีและปลอดภัยที่จะรันกับอะไรก็ได้

```sql
EXPLAIN SELECT * FROM books WHERE author_id = 4;
```

```text
                              QUERY PLAN
----------------------------------------------------------------------
 Index Scan using books_author_idx on books  (cost=0.42..12.61 rows=6 width=64)
   Index Cond: (author_id = 4)
```

อ่านตัวเลขในวงเล็บ:

- `cost=0.42..12.61` — สองตัวเลขในหน่วยต้นทุนแบบสมมติ ตัวแรกคือต้นทุนเริ่มต้นโดยประมาณ (งานก่อนที่แถวแรกจะปรากฏ) ส่วนตัวที่สองคือต้นทุนรวมโดยประมาณ ค่ายิ่งต่ำยิ่งถูก planner จะนำค่าเหล่านี้มาเปรียบเทียบกันระหว่างแผนที่เป็นตัวเลือก
- `rows=6` — จำนวนแถวที่ planner *ประมาณ* ว่าโหนดนี้จะคืนกลับมา
- `width=64` — ขนาดเฉลี่ยของแถวโดยประมาณในหน่วยไบต์

บรรทัดบนสุดยังระบุชื่อกลยุทธ์ด้วย ในที่นี้คือ `Index Scan` ที่ใช้ index ที่มีชื่อ โดยมี `Index Cond` แสดงว่า index จัดการเงื่อนไขใด

## EXPLAIN ANALYZE: สิ่งที่เกิดขึ้นจริง

`EXPLAIN ANALYZE` ทำการ *รัน* query จริง ๆ และรายงานเวลาที่เกิดขึ้นจริงควบคู่ไปกับค่าประมาณ การเพิ่ม `BUFFERS` ยังรายงานด้วยว่าอ่านหน้าข้อมูล (data page) ไปกี่หน้า ซึ่งบอกคุณได้ว่างานมาจากแคชหรือจากดิสก์

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM books WHERE author_id = 4;
```

```text
                                                  QUERY PLAN
--------------------------------------------------------------------------------------------------------------
 Index Scan using books_author_idx on books  (cost=0.42..12.61 rows=6 width=64) (actual time=0.018..0.024 rows=6 loops=1)
   Index Cond: (author_id = 4)
   Buffers: shared hit=4
 Planning Time: 0.092 ms
 Execution Time: 0.041 ms
```

บล็อก `actual time=0.018..0.024 rows=6 loops=1` ใหม่นี้คือความจริง: เวลาเริ่มต้นและเวลารวมจริงในหน่วยมิลลิวินาที, จำนวนแถวจริง และจำนวนครั้งที่โหนดนี้ทำงาน จงเปรียบเทียบ `rows` ที่ประมาณกับ `rows` จริง: ช่องว่างที่กว้างคือสัญญาณคลาสสิกของ statistics ที่เก่าล้าสมัยหรือ planner ที่ถูกชี้นำผิดทาง `Buffers: shared hit=4` หมายความว่ามีสี่หน้าที่ถูกเสิร์ฟมาจากแคชและไม่มีหน้าใดมาจากดิสก์

## การแยกแยะชนิดของ scan

กลยุทธ์ของ scan สามแบบปรากฏอยู่ตลอดเวลา การเรียนรู้ที่จะจดจำพวกมันคือส่วนใหญ่ของการต่อสู้

- **Seq Scan** — อ่านทั้งตารางตั้งแต่บนลงล่าง คาดว่าจะเกิดขึ้นเมื่อไม่มี index ที่เป็นประโยชน์อยู่ หรือเมื่อ query คืนแถวเป็นสัดส่วนใหญ่ของตารางอยู่แล้ว
- **Index Scan** — เดินผ่าน index แล้วจึงดึงแต่ละแถวที่ตรงเงื่อนไขจากตาราง ดีที่สุดเมื่อ query นั้น *เลือกเฉพาะเจาะจง* คือคืนข้อมูลเพียงส่วนน้อยของตาราง
- **Bitmap Heap Scan** (จับคู่กับ **Bitmap Index Scan**) — สร้าง bitmap ของตำแหน่งแถวที่ตรงเงื่อนไขจาก index, เรียงลำดับมัน แล้วจึงอ่านตารางตามลำดับทางกายภาพ PostgreSQL เลือกทางสายกลางนี้เมื่อ query ตรงกับแถวมากกว่าที่ index scan จัดการได้ดี แต่น้อยกว่า seq scan แบบเต็ม

แผนแบบ bitmap มีหน้าตาแบบนี้:

```text
                                  QUERY PLAN
------------------------------------------------------------------------------
 Bitmap Heap Scan on books  (cost=12.10..520.30 rows=900 width=64)
   Recheck Cond: (published > 2000)
   ->  Bitmap Index Scan on books_published_idx  (cost=0.00..11.88 rows=900 width=0)
         Index Cond: (published > 2000)
```

บรรทัด `->` ที่ย่อหน้าเข้าไปคือโหนดลูกที่ป้อนข้อมูลให้โหนดแม่ของมัน แผนเป็นโครงสร้างต้นไม้ (tree): จงอ่านจากโหนดที่ย่อหน้ามากที่สุดออกมาด้านนอก เพราะโหนดด้านในทำงานก่อนและส่งแถวของมันขึ้นไป

## ทำไมบางครั้ง planner จึงเพิกเฉยต่อ index

เรื่องน่าประหลาดใจที่พบบ่อย: คุณสร้าง index แต่ `EXPLAIN` ก็ยังแสดง `Seq Scan` อยู่ดี ปกติแล้วนี่คือ planner ที่ *ถูกต้อง* ไม่ใช่ผิด:

- **query ไม่ได้เลือกเฉพาะเจาะจง** หากเงื่อนไขตรงกับครึ่งหนึ่งของตาราง การกระโดดไปกลับระหว่าง index กับตารางจะช้ากว่าการอ่านตารางทื่อ ๆ ไปเรื่อย ๆ seq scan จึงชนะ
- **ตารางเล็กมาก** บนตารางที่มีเพียงไม่กี่ร้อยแถว seq scan แทบจะไม่มีต้นทุน และ planner ก็จะข้ามภาระของ index ไป
- **statistics เก่าล้าสมัย** หาก planner คิดว่าเงื่อนไขหนึ่งตรงกับแถวมากกว่าความเป็นจริงไปมาก มันอาจหลีกเลี่ยง index การรัน `ANALYZE` บนตารางจะแก้ไขค่าประมาณให้ถูกต้อง
- **เงื่อนไขไม่สามารถใช้ index ได้** ทั้ง `LIKE` แบบ leading-wildcard, ฟังก์ชันที่ครอบคอลัมน์ไว้ หรือชนิดข้อมูลที่ไม่ตรงกัน ล้วนทำให้ index ที่มีอยู่ใช้งานไม่ได้

**ใน pgAdmin:** ปุ่ม Explain รัน `EXPLAIN` และเมนู Explain มีตัวเลือก Analyze สำหรับ `EXPLAIN ANALYZE` ผลลัพธ์จะถูกวาดออกมาเป็นต้นไม้แบบกราฟิก โดยแต่ละโหนดจะแสดงชนิดของ scan และต้นทุน และลูกศรที่หนากว่าหมายถึงแถวที่มากกว่า — เป็นวิธีที่รวดเร็วในการมองหาโหนดที่แพง

## Tips / gotchas

- `EXPLAIN ANALYZE` ทำการรัน query จริง ๆ สำหรับ `SELECT` นั้นไม่เป็นอันตราย แต่สำหรับ `UPDATE` หรือ `DELETE` มันจะเปลี่ยนข้อมูลของคุณ — จงห่อมันไว้ใน transaction ที่คุณ `ROLLBACK` หากคุณจำเป็นต้องตรวจสอบการเขียน
- แผนจะมีความหมายก็ต่อเมื่ออยู่บนข้อมูลที่สมจริงเท่านั้น query ที่ทำ seq scan บนตารางทดสอบ 100 แถว อาจทำ index scan บนตารางเดียวกันที่มีล้านแถว จงทดสอบกับปริมาณข้อมูลที่เป็นตัวแทนจริง
- จงเปรียบเทียบ `rows` ที่ประมาณกับ `rows` จริงเสมอ ความไม่ตรงกันที่มากชี้ไปยัง statistics ที่ต้องการการรีเฟรช
- จงอ่านต้นไม้ของแผนจากในออกนอก: โหนดที่ย่อหน้ามากที่สุดทำงานก่อนและป้อนข้อมูลให้โหนดแม่ของมัน
