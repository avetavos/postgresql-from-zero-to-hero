query ที่คืนคำตอบถูกต้องก็ยังเป็น query ที่ช้าได้ เมื่อคุณขอให้ MongoDB หาสมาชิกทุกคนที่เข้าร่วมในปี 2023 และไม่มีอะไรช่วยมันเลย เซิร์ฟเวอร์มีทางเลือกเดียว นั่นคือเปิด collection ที่เอกสารแรกแล้วอ่านไปข้างหน้า ตรวจทีละตัว จนกว่าจะดูครบทุกเอกสาร การอ่านแบบกำลังดิบนี้เรียกว่า **collection scan** และ MongoDB ติดป้ายมันว่า `COLLSCAN` ในข้อมูลวินิจฉัย กับสมาชิกห้องสมุดไม่กี่คนมันมองไม่เห็น แต่บน collection ที่มีเอกสารสิบล้านรายการ มันคือความต่างระหว่างการตอบกลับในหนึ่งมิลลิวินาทีกับการตอบกลับที่คลานต้วมเตี้ยม

**index** คือทางแก้ มันเป็นโครงสร้างข้อมูลแยกต่างหากที่เรียงลำดับไว้ ซึ่งเซิร์ฟเวอร์ดูแลควบคู่ไปกับ collection ของคุณ เก็บค่าของฟิลด์หนึ่งหรือหลายฟิลด์ในลำดับที่เรียงไว้ โดยแต่ละค่าจับคู่กับพอยน์เตอร์ที่ชี้กลับไปยังเอกสารฉบับเต็ม เพราะค่าถูกเรียงไว้ เซิร์ฟเวอร์จึงกระโดดตรงไปยังค่าที่ต้องการได้แทนที่จะอ่านทั้งหมด ด้วยเหตุผลเดียวกับที่คุณพลิกไปท้ายเล่มตำราแทนที่จะอ่านทุกหน้าเพื่อหาว่าคำศัพท์หนึ่งถูกนิยามไว้ตรงไหน ทั้งโมดูลนี้ว่าด้วยการสร้าง index ที่ถูกต้องสำหรับ query ที่ถูกต้อง และพิสูจน์ชัยชนะด้วย `explain`

## scan เทียบกับ seek

ลองนึกภาพสองกลยุทธ์วางเทียบกัน เมื่อไม่มี index นั้น query จะเดินผ่านทั้ง collection เมื่อมี index มันจะไต่ลงต้นไม้ที่เรียงลำดับไว้แล้วลงตรงช่วงที่ตรงกันพอดี:

เส้นทางด้านขวาแตะเฉพาะเอกสารที่ต้องการ ส่วนเส้นทางด้านซ้ายแตะทุกตัวแล้วโยนทิ้งเกือบหมด การอ่านที่สูญเปล่านั้นแหละคือสิ่งที่ index ขจัดออกไป

## index แรก

การสร้าง index เป็นการเรียกครั้งเดียว คุณตั้งชื่อฟิลด์และทิศทาง — `1` สำหรับจากน้อยไปมาก — แล้วเซิร์ฟเวอร์จะสร้างโครงสร้างให้คุณ ตรงนี้เราทำ index บนฟิลด์ `joined` ของ `members` ในห้องสมุด เพื่อให้การกรองหรือเรียงตามปีที่เข้าร่วมกลายเป็นการ seek:

การเรียกนั้นคืนชื่อที่ MongoDB ตั้งให้ index ใหม่ — มันอนุมานชื่อจากฟิลด์และทิศทางเว้นแต่คุณจะกำหนดเอง:

```json
{ "createdIndexes": ["joined_1"], "ok": 1 }
```

**ใน Compass:** เปิด collection แล้วเลือกแท็บ **Indexes** คุณจะเห็น index `_id_` ในตัวอยู่เสมอ และคุณเพิ่มอันใหม่ได้ด้วยปุ่ม **Create Index** โดยพิมพ์ชื่อฟิลด์และเลือกทิศทาง

## โมดูลนี้ครอบคลุมอะไรบ้าง

บทเรียนต่อยอดจากฟิลด์เดียวไปสู่กล่องเครื่องมือเต็มชุด:

1. **ทำไม index จึงสำคัญ** — scan เทียบกับ seek และต้นทุนของการไม่มีเลย
2. **single และ compound indexes** — การทำ index ฟิลด์เดียว แล้วหลายฟิลด์ และ prefix rule ที่ตัดสินว่า compound index ให้บริการ query ใดได้บ้าง
3. **กฎ ESR** — การจัดลำดับฟิลด์ (Equality, Sort, Range) ที่ทำให้ compound index คุ้มค่ากับที่ลงทุนไป
4. **explain และ query plan** — การอ่านผลลัพธ์ของ `explain` เพื่อยืนยันว่ามีการใช้ index และจับ covered query
5. **ประเภทของ index** — multikey, text, TTL, partial, unique และ sparse index สำหรับกรณีที่ index ธรรมดาครอบคลุมไม่ได้

## เคล็ดลับและจุดพลาด

- ทุก collection เริ่มต้นด้วย index หนึ่งอันอยู่แล้ว นั่นคือ unique index บน `_id` คุณไม่ต้องสร้างมันเลย และดรอปมันไม่ได้
- index ไม่ได้มาฟรี มันเร่งการอ่านให้เร็วขึ้นแต่เพิ่มงานให้ทุก insert, update และ delete เพราะ index ต้องถูกทำให้ตรงกันตลอด ทำ index เฉพาะฟิลด์ที่ query ของคุณใช้จริง ไม่ใช่ทุกฟิลด์
- index อยู่บนดิสก์และในหน่วยความจำเหมือนกับข้อมูลที่มันบรรยายถึง กอง index ที่ไม่ได้ใช้ก็เปลืองทั้งสองอย่าง จึงคุ้มที่จะลบอันที่ไม่มี query ใดพึ่งพาออกไป

<Callout title="หนึ่งแนวคิดเบื้องหลัง index ทุกอัน">
index แลกต้นทุนการเขียนที่เพิ่มขึ้นเล็กน้อยและพื้นที่จัดเก็บ กับความสามารถในการหาเอกสารด้วยการ seek ผ่านค่าที่เรียงลำดับไว้แทนที่จะสแกนทุกเอกสาร ส่วนที่เหลือของโมดูลนี้เป็นเพียงการเลือกว่าจะเรียงฟิลด์ใด
</Callout>
