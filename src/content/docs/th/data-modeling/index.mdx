ถ้าคุณมา MongoDB จากพื้นฐานเชิงสัมพันธ์ (relational) นิสัยที่เลิกยากที่สุดคือสิ่งที่เคยช่วยคุณได้ดีที่สุดใน SQL นั่นคือการแยกข้อมูลออกเป็นหลาย ๆ ตารางแคบ ๆ แต่ละตารางเก็บของอย่างหนึ่ง แล้วค่อย join กลับมารวมกันตอน query วินัยแบบนี้เรียกว่า **normalization** และมันมีขึ้นเพื่อกำจัดความซ้ำซ้อน เพื่อให้ข้อเท็จจริงหนึ่งอยู่ในที่เดียวเท่านั้น MongoDB ขอให้คุณคิดกลับด้าน คุณไม่ได้เริ่มจาก "entity ของฉันมีอะไรบ้าง และจะหลีกเลี่ยงการเก็บซ้ำได้อย่างไร" แต่เริ่มจาก "แอปพลิเคชันของฉันถามหาอะไร และการอ่านครั้งเดียวจะคืนทุกอย่างที่หน้าจอหนึ่งต้องการได้อย่างไร"

หน่วยของการจัดเก็บคือ **document** — โครงสร้างแบบ JSON ที่อุดมและซ้อนกันได้ ซึ่งสามารถเก็บ array และ sub-object ได้ ไม่ใช่แค่ column แบน ๆ เพราะ document สามารถเก็บข้อมูลที่เกี่ยวข้องไว้ในตัวมันเองได้ คำถามที่ครอบงำทุกการตัดสินใจในการ modeling จึงไม่ใช่ "column นี้ควรอยู่ตารางไหน" อีกต่อไป แต่เป็น **"ฉันจะ nest ข้อมูลที่เกี่ยวข้องนี้ไว้ใน document หรือเก็บมันแยกไว้แล้วเก็บ reference ชี้ไปหา?"** ทางเลือกเดียวนี้ — embedding กับ referencing — คือสิ่งที่ทั้งโมดูลนี้หมุนรอบ

## นิสัยเชิงสัมพันธ์ กับการคิดแบบ document

ใน schema เชิงสัมพันธ์ คุณออกแบบตามรูปร่างของข้อมูลแล้วปล่อยให้ query จ่ายต้นทุนในการประกอบมันกลับคืน ส่วนใน document schema คุณออกแบบตาม query แล้วปล่อยให้รูปร่างของข้อมูลรับต้นทุนไว้ล่วงหน้า สองแนวคิดนี้ดึงไปคนละทิศทาง:

ไม่มีแนวทางใด "ถูกต้อง" เสมอไป สัญชาตญาณเชิงสัมพันธ์ยอดเยี่ยมเมื่อข้อมูลมีความเกี่ยวพันกันสูงและถูกอัปเดตจากหลายทิศทาง ส่วนสัญชาตญาณแบบ document โดดเด่นเมื่อแอปพลิเคชันดึงข้อมูลกลุ่มที่มีรูปร่างเดิม ๆ ซ้ำ ๆ — ผู้ใช้กับการตั้งค่าของเขา คำสั่งซื้อกับรายการสินค้าในนั้น บทความกับ tag ของมัน MongoDB ให้คุณเก็บกลุ่มข้อมูลนั้นในแบบที่แอปพลิเคชันต้องการนำไปใช้

## ตัวอย่างจริง: คำสั่งซื้อ

ลองพิจารณาคำสั่งซื้อออนไลน์ ระบบเชิงสัมพันธ์จะเก็บคำสั่งซื้อไว้ตารางหนึ่ง รายการสินค้าไว้อีกตาราง และลูกค้าไว้ตารางที่สาม แล้วเย็บเข้าด้วยกันด้วย foreign key ส่วนใน MongoDB คุณสามารถเก็บคำสั่งซื้อทั้งก้อน — โดย embed รายการสินค้าเป็น array — ไว้ใน document เดียว ทำให้การอ่านคำสั่งซื้อเป็นการทำงานครั้งเดียวแทนที่จะต้อง join สามครั้ง:

```json
{
  "_id": "order_5001",
  "customer": "Grace Hopper",
  "placedAt": "2026-03-14T09:12:00Z",
  "status": "shipped",
  "items": [
    { "sku": "BK-204", "title": "Distributed Systems", "qty": 1, "price": 42.0 },
    { "sku": "BK-991", "title": "The Mythical Man-Month", "qty": 2, "price": 18.5 }
  ],
  "total": 79.0
}
```

document หนึ่งนั้นคือทุกอย่างที่หน้า "รายละเอียดคำสั่งซื้อ" ต้องการ ไม่มี join ไม่มีการวิ่งไปกลับครั้งที่สอง — ข้อมูลถูกจัดรูปร่างให้เป็นคำตอบอยู่แล้ว ข้อแลกเปลี่ยนคือ ตอนนี้ชื่อและราคาของสินค้ากลายเป็นสำเนาของข้อมูลที่อาจอยู่ใน `products` collection ด้วย และสำเนาก็คลาดเคลื่อนได้ การตัดสินว่าเมื่อใดการแลกเปลี่ยนนี้คุ้มค่า คือศิลปะของการ data modeling และเนื้อหาที่เหลือของโมดูลนี้จะให้เครื่องมือแก่คุณในการตัดสินใจได้ดี

## โมดูลนี้ครอบคลุมอะไรบ้าง

บทเรียนต่าง ๆ ต่อยอดจากทางเลือกหลักออกไปสู่โครงสร้างที่นำกลับมาใช้ซ้ำได้:

1. **Embedding กับ referencing** — สองวิธีในการเชื่อมโยง document เข้าด้วยกัน และข้อแลกเปลี่ยนเรื่อง read locality, ความซ้ำซ้อน และการเติบโตของ document
2. **Modeling relationships** — one-to-one, one-to-many และ many-to-many รวมถึงข้อจำกัด document ขนาด 16 MB และอันตรายของ array ที่ไม่มีขอบเขต
3. **Schema design patterns** — Subset, Computed, Bucket และ Extended Reference pattern ที่แก้ปัญหาที่เกิดซ้ำ ๆ
4. **Schema validation** — ทำให้ schema เป็นแบบ optional-but-enforced ด้วยกฎ `$jsonSchema`
5. **Anti-patterns** — ความผิดพลาดในการ modeling ที่ดูเหมือนไม่มีปัญหาตอนแรกแต่เจ็บตอนสเกล และวิธีแก้แต่ละข้อ

## เคล็ดลับและข้อควรระวัง

- MongoDB เป็นแบบ **schema-flexible** ไม่ใช่ schema-less document ใน collection เดียวกันแตกต่างกันได้ แต่รูปร่างที่ตั้งใจและสม่ำเสมอคือสิ่งที่ทำให้ collection เร็วและดูแลรักษาได้
- "Model for your queries" หมายความว่าคุณควรรู้ access pattern หลักของคุณ *ก่อน* ออกแบบ schema ที่สมบูรณ์แบบสำหรับการอ่านรูปแบบหนึ่งอาจผิดสำหรับอีกรูปแบบหนึ่ง
- Embedding ไม่ได้ดีกว่า referencing เสมอ และ referencing ก็ไม่ได้ "ถูกต้อง" กว่าเสมอ ทุกบทเรียนที่นี่คือการอ่านข้อแลกเปลี่ยน ไม่ใช่การท่องจำกฎ
- ความซ้ำซ้อนเป็นที่ยอมรับได้ และบางครั้งก็เป็นที่พึงปรารถนาใน MongoDB ต้นทุนของสำเนาคือการคอยทำให้มันตรงกัน ส่วนประโยชน์คือการอ่านที่เร็วและง่ายกว่า
