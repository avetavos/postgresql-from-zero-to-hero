การเขียน SQL ให้ถูกต้องเป็นเพียงครึ่งหนึ่งของงาน อีกครึ่งหนึ่งคือการทำให้เซิร์ฟเวอร์ PostgreSQL เพียงตัวเดียวยังทำงานได้ดีเมื่อต้องให้บริการผู้ใช้หลายพันคนในทันที รอดจากดิสก์ที่พังลง รองรับตารางที่มีข้อมูลหลายพันล้านแถว และปฏิเสธทุกคนที่ไม่ควรเข้ามา กลุ่มของเรื่องเหล่านี้คือสิ่งที่คนเรียกว่า **operations** — งานของการรันฐานข้อมูลในโปรดักชันแทนที่จะรันบนแล็ปท็อปของคุณ

โมดูลนี้ก้าวออกจากไวยากรณ์ของ query แล้วมองไปที่รูปร่างของการ deploy จริง ไม่มีอะไรในนี้เปลี่ยน SQL ที่คุณรู้อยู่แล้ว แต่กลับเป็นการล้อมเซิร์ฟเวอร์ตัวเดียวนั้นด้วยชิ้นส่วนที่ทำให้มันน่าเชื่อถือเมื่อต้องขยายขนาด

## เรื่องสำคัญสี่ด้านในโปรดักชัน

แทบทุกหัวข้อด้าน operations ใน PostgreSQL จัดอยู่ภายใต้หนึ่งในสี่หัวข้อนี้ แต่ละหัวข้อตอบคำถามแบบ "จะเกิดอะไรขึ้นเมื่อ..." ที่แตกต่างกัน

| เรื่องสำคัญ           | คำถามที่มันตอบ                                    | คำตอบที่มักใช้กัน           |
|----------------------|--------------------------------------------------|-----------------------------|
| High availability    | จะเกิดอะไรขึ้นถ้าเซิร์ฟเวอร์ primary ตาย?         | Replication และ failover    |
| Client จำนวนมาก      | จะเกิดอะไรขึ้นถ้าแอปหลายพันตัวเชื่อมต่อพร้อมกัน?  | Connection pooling          |
| ตารางขนาดใหญ่        | จะเกิดอะไรขึ้นถ้าตารางหนึ่งโตถึงหลายพันล้านแถว?   | Partitioning                |
| Durability และ access | จะเกิดอะไรขึ้นถ้าดิสก์พังหรือมีผู้โจมตีเชื่อมต่อ? | Backup และ security         |

โปรเจกต์เล็ก ๆ อาจมองข้ามทั้งสี่ด้านได้ แต่ระบบโปรดักชันที่จริงจังในที่สุดก็ต้องการทุกด้าน และทั้งสี่ด้านนี้มีปฏิสัมพันธ์กัน: replica ช่วยทั้งเรื่องความพร้อมใช้งาน *และ* กระจายภาระการอ่าน, pooling ปกป้อง primary ที่ป้อนข้อมูลให้ replica อยู่ด้วย และอื่น ๆ อีกมากมาย

## การ deploy ในรูปแบบโปรดักชัน

แล็ปท็อปรัน PostgreSQL หนึ่ง process กับ client หนึ่งตัว แต่การตั้งค่าในโปรดักชันกระจายงานออกไปยังชิ้นส่วนหลายชิ้น แอปพลิเคชันเซิร์ฟเวอร์ไม่ได้คุยกับฐานข้อมูลโดยตรง แต่ผ่าน **connection pooler** ที่คอยรักษาเซิร์ฟเวอร์ connection จำนวนน้อยที่นำกลับมาใช้ซ้ำได้ มีโหนดหนึ่งคือ **primary** ที่รับงานเขียน และมี **replica** หนึ่งตัวหรือมากกว่านั้นที่รับสำเนาการเปลี่ยนแปลงอย่างต่อเนื่องและตอบ query แบบอ่านอย่างเดียว

อ่านไดอะแกรมจากด้านบน แอปเซิร์ฟเวอร์จำนวนมากถูกรวมผ่าน pooler ตัวเดียว เพื่อให้ primary ไม่ต้องเจอกับ connection ดิบจำนวนมหาศาล primary ส่ง stream ของทุกการเปลี่ยนแปลงไปยัง replica ของมัน ซึ่ง replica เหล่านั้นรับภาระการอ่านออกจากหลังของ primary และข้อมูลของ primary ก็ถูกคัดลอกไปยัง backup storage อย่างต่อเนื่อง เพื่อให้สามารถสร้างทั้งระบบขึ้นมาใหม่ได้หลังจากภัยพิบัติ

## โมดูลนี้ครอบคลุมอะไรบ้าง

- **Replication** — การ stream write-ahead log ของ primary ไปยัง read replica เพื่อความพร้อมใช้งาน และ logical replication สำหรับคัดลอกตารางที่เลือกระหว่างเซิร์ฟเวอร์
- **Connection pooling** — ทำไม PostgreSQL connection แต่ละตัวจึงเป็น process ของระบบปฏิบัติการ และ pooler อย่าง PgBouncer ทำ multiplex client จำนวนมากเข้ากับเซิร์ฟเวอร์ connection จำนวนน้อยได้อย่างไร
- **Partitioning** — การแบ่งตารางเชิงตรรกะขนาดมหึมาหนึ่งตารางออกเป็น partition ทางกายภาพจำนวนมาก เพื่อให้ query และงาน maintenance แตะเฉพาะส่วนที่ต้องการ
- **Backup และ security** — backup แบบ logical และ physical, point-in-time recovery, role และ `GRANT`, row-level security, และการล็อกว่าใครเชื่อมต่อได้บ้าง

## สิ่งนี้อยู่ตรงไหน

คุณรู้วิธีออกแบบโมเดลข้อมูล query มัน สร้าง index ให้มัน และห่อหุ้มการเปลี่ยนแปลงไว้ใน transaction แล้ว Operations คือสิ่งที่ทำให้ความรู้เหล่านั้นรอดจากการปะทะกับ traffic จริง ไม่มีอะไรในนี้เป็น PostgreSQL คนละตัว มันคือเซิร์ฟเวอร์ตัวเดิม ที่ถูกตั้งค่าและล้อมรอบไว้เพื่อให้ขีดจำกัดของเครื่องเดียวและความล้มเหลวของเครื่องเดียวไม่กลายเป็นขีดจำกัดและความล้มเหลวของแอปพลิเคชันคุณ

## เคล็ดลับ / ข้อควรระวัง

- อย่าเพิ่งคว้า replication หรือ partitioning มาใช้ตั้งแต่วันแรก เซิร์ฟเวอร์เดียวที่ปรับแต่งดีพร้อม index ที่ดีรองรับภาระได้มากอย่างน่าประหลาดใจ เพิ่มความซับซ้อนก็ต่อเมื่อคุณวัดความจำเป็นได้แล้วเท่านั้น
- Replica มีไว้เพื่อความพร้อมใช้งานและการขยายการอ่าน ไม่ใช่เพื่อรับงานเขียน — ทุกการเขียนยังคงไปที่ primary ตัวเดียว
- Backup ที่คุณไม่เคย restore เลยคือความหวัง ไม่ใช่ backup ข้อพิสูจน์เดียวคือการทดสอบ restore ที่สำเร็จ
- Security ไม่ใช่ขั้นตอนสุดท้ายที่คุณค่อยติดเข้าไป role แบบ least-privilege และการจำกัดการเข้าถึงเครือข่ายตั้งค่าได้ถูกที่สุดก่อนที่จะมีอะไรพึ่งพาเวอร์ชันที่หละหลวม

Operations ไม่ได้มาแทนสิ่งที่คุณเรียนรู้มา — มันขยายมัน primary ในทุกไดอะแกรมของโมดูลนี้กำลังรัน SQL engine ตัวเดียวกันกับที่คุณใช้ในบทเรียนก่อนหน้า Replication, pooling, partitioning, และ backup คือตัวประกอบที่ทำให้ engine นั้นแสดงผลได้ในโปรดักชัน
