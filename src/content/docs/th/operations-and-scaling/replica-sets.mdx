replica set คือคำตอบของคำถามตรง ๆ ข้อหนึ่ง: จะเกิดอะไรขึ้นเมื่อเซิร์ฟเวอร์ที่ถือข้อมูลของคุณหยุดตอบสนอง? เมื่อมี `mongod` ตัวเดียว คำตอบคือแอปพลิเคชันของคุณจะล่มจนกว่าจะมีคนแก้เครื่องนั้น เมื่อมี replica set คำตอบคือสำเนาอีกชุดจะเข้ามาทำหน้าที่แทนภายในไม่กี่วินาที และผู้ใช้ส่วนใหญ่จะไม่รู้สึกถึงมันเลย **replica set** คือกลุ่มของ `mongod` instances ที่ถือข้อมูลชุดเดียวกันทั้งหมด — หนึ่ง **primary** ที่รับทุก write และหนึ่งหรือมากกว่า **secondaries** ที่เก็บสำเนาของตัวเองให้ตามทันมัน

## Primary, secondaries และ oplog

มีแต่ primary เท่านั้นที่รับ write แต่ละ write ที่มันทำยังถูกบันทึกลงใน capped collection พิเศษที่เรียกว่า **oplog** — operations log ทุก secondary อ่าน oplog ของ primary อย่างต่อเนื่องและเล่นซ้ำ operations เหล่านั้นกับข้อมูลของตัวเอง เพื่อให้สำเนาบรรจบกัน นี่คือเหตุผลที่ secondaries เป็นแบบ *eventually* consistent: มี lag เล็กน้อยระหว่างที่ write ลงบน primary กับที่ write เดียวกันนั้นถูกเล่นซ้ำในที่อื่น ๆ ทั้งหมด

สมาชิกยังแลกเปลี่ยนข้อความ **heartbeat** เล็ก ๆ กันสองสามครั้งต่อนาที heartbeat เหล่านี้คือวิธีที่ set สังเกตเห็นปัญหา: ถ้า secondaries หยุดได้ยินเสียงจาก primary พวกมันจะสรุปว่ามันล้มเหลวและเริ่ม election

## Election อัตโนมัติ

เมื่อ primary หายไป สมาชิกที่เหลือจะลงคะแนนเสียง secondary ที่มีสิทธิ์จะเสนอชื่อตัวเอง คนอื่นลงคะแนน และผู้สมัครที่ได้คะแนนเสียงข้างมากจะกลายเป็น primary ใหม่ นี่คือหัวใจของ high availability — มันเกิดขึ้นโดยอัตโนมัติ โดยปกติภายในประมาณสิบถึงสิบสองวินาที โดยไม่มีใครต้องถูกตามตัวตอนตีสาม

<Mermaid code={electionFlow} title="election เลื่อนตำแหน่ง secondary ขึ้นเป็น primary เพื่อให้ write ดำเนินต่อได้โดยไม่ต้องแทรกแซงด้วยมือ" />

คำว่า **majority** ทำงานสำคัญตรงนี้ ผู้สมัครต้องเก็บคะแนนเสียงจากสมาชิกที่มีสิทธิ์ลงคะแนนมากกว่าครึ่งหนึ่ง ข้อกำหนดนั้นคือเหตุผลของคำแนะนำเรื่อง replica set ที่ถูกพูดซ้ำบ่อยที่สุด: ให้มีจำนวนสมาชิกที่ลงคะแนนเป็น **เลขคี่** ด้วยสามเสียง การล้มเหลวเพียงครั้งเดียวก็ยังเหลือเสียงข้างมากที่ชัดเจนคือสองเสียง; ด้วยสองเสียง การล้มเหลวเพียงครั้งเดียวจะทำให้เสมอกัน และไม่มีใครชนะได้

## การเชื่อมต่อกับ replica set

driver ไม่ได้เชื่อมต่อกับเซิร์ฟเวอร์ตายตัวหนึ่งตัว — มันเชื่อมต่อกับ *set* คุณระบุสมาชิกสองสามตัวเป็น seeds และตั้งชื่อ set ด้วย `replicaSet=` จากนั้น driver ค้นพบ topology ทั้งหมดเอง หา primary ปัจจุบันได้ด้วยตัวมันเอง และตาม primary โดยอัตโนมัติเมื่อข้าม election

โดยค่าเริ่มต้น การอ่านก็ไปที่ primary เช่นกัน ซึ่งทำให้มัน strongly consistent ถ้าคุณยอมรับความล้าสมัยเล็กน้อยได้ — เช่น สำหรับ analytics หรือ feed ที่ไม่จำเป็นต้องได้ write ล่าสุดที่สุด — คุณสามารถบอก driver ให้อ่านจาก secondaries ด้วย read preference เช่น `secondaryPreferred` เพื่อกระจายภาระการอ่านออกจาก primary

## การตรวจสอบ set

จาก `mongosh` ที่เชื่อมต่ออยู่กับสมาชิกตัวใดก็ได้ `rs.status()` รายงานสถานะสด ๆ ของทั้ง set: ใครเป็น primary, ใครเป็น secondary และสมาชิกแต่ละตัวตามอยู่ห่างแค่ไหน นี่คือคำสั่งแรกที่คุณรันเมื่อมีอะไรดูผิดปกติ

```js
rs.status()
```

คำตอบที่ตัดทอนแล้วแสดงสมาชิกแต่ละตัวพร้อมบทบาทของมันในฟิลด์ `stateStr`:

```json
{
  "set": "rs0",
  "members": [
    { "name": "host1:27017", "stateStr": "PRIMARY", "health": 1 },
    { "name": "host2:27017", "stateStr": "SECONDARY", "health": 1 },
    { "name": "host3:27017", "stateStr": "SECONDARY", "health": 1 }
  ],
  "ok": 1
}
```

**ใน Atlas:** ทุก cluster เป็น replica set อยู่แล้ว — คุณไม่ต้องตั้งค่าด้วยมือ มุมมอง cluster แสดงว่า node ไหนเป็น primary และให้คุณดู replication lag บนกราฟได้ และ failover เป็นการทดสอบแบบคลิกเดียวที่คุณ trigger ได้เพื่อดู election เกิดขึ้น

## เคล็ดลับและข้อควรระวัง

- ให้มีจำนวนสมาชิกที่ลงคะแนนเป็น **เลขคี่** เพื่อให้การล้มเหลวครั้งเดียวเหลือเสียงข้างมากที่ชัดเจนเสมอ สามตัวคือขั้นต่ำที่นิยมใช้สำหรับ production
- **arbiter** คือสมาชิกขนาดเบาที่ลงคะแนนแต่ไม่ถือข้อมูล มันสามารถตัดสินกรณีเสมอได้อย่างประหยัด แต่มันกลายเป็น primary ไม่ได้และไม่เพิ่มความทนทาน ดังนั้นให้เลือก secondary จริงที่ถือข้อมูลเมื่อคุณมีพื้นที่จัดเก็บพอ
- secondaries มี lag ถ้าแอปพลิเคชันของคุณอ่าน write ของตัวเองทันทีหลังจากเขียน ให้อ่านจาก primary หรือใช้ read concern ที่รอให้ write ทำ replication เสร็จ ไม่เช่นนั้นคุณอาจยังไม่เห็นการเปลี่ยนแปลงของตัวเอง
- replica set ป้องกัน node ที่ล้มเหลว ไม่ได้ป้องกันคิวรีที่ผิดหรือ collection ที่ถูกลบทิ้ง — สิ่งเหล่านั้นถูก replicate ตามไปด้วย replication คือ availability ไม่ใช่ backup
