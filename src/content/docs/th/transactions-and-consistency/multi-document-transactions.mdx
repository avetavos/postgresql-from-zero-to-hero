บางครั้งการดำเนินการครั้งเดียวจำเป็นต้องเปลี่ยนแปลงหลายเอกสารจริง ๆ และต้องสำเร็จทั้งหมดหรือไม่สำเร็จเลยสักอย่าง กรณีในตำราเรียนคือการย้ายเงิน: หักจากบัญชีหนึ่งและเพิ่มให้อีกบัญชีหนึ่ง ถ้าการหักลงแต่การเพิ่มล้มเหลว เงินก็หายไป เมื่อเอกสารที่เกี่ยวข้องไม่สามารถรวมเป็นหนึ่งได้ — เพราะมันเป็นคนละบัญชี หรืออยู่คนละคอลเลกชัน — MongoDB มี **multi-document transaction** จริง ๆ ให้ มันให้การรับประกัน ACID เต็มรูปแบบข้ามทุกการดำเนินการภายในมัน: ทั้งหมด commit ด้วยกันหรือ roll back ด้วยกัน

ราคาของการรับประกันนั้นคือ transaction ต้องใช้ **session** เพื่อรัน ต้องใช้ **replica set** (standalone server รันมันไม่ได้) และมีต้นทุนสูงกว่าการเขียนธรรมดา ดังนั้นนี่จึงเป็นเครื่องมือที่ใช้อย่างจงใจ ไม่ใช่ค่าเริ่มต้น

## transaction อยู่ใน session

ขั้นตอนเป็นแบบเดิมเสมอ คุณเริ่ม **session** จากนั้นรันการดำเนินการของคุณภายใน transaction ที่ผูกกับ session นั้น ตัวช่วยของ driver ที่เรียกว่า `withTransaction` ห่อหุ้มทุกอย่างไว้: มันเริ่ม transaction รัน callback ของคุณ **commit** ถ้ามันคืนค่ามาอย่างสะอาด และ **abort** (roll back ทุกอย่าง) ถ้ามัน throw นอกจากนี้มันยังลอง commit ใหม่โดยอัตโนมัติเมื่อเกิด transient error บางอย่าง ซึ่งเป็นเหตุผลที่คุณควรใช้มันมากกว่าการเริ่มและ commit ด้วยมือ

## การโอนเงินระหว่างบัญชี

ตรงนี้เราย้ายค่าสามหน่วยจาก Ada ไปยัง Linus ข้ามสองเอกสารในคอลเลกชัน `accounts` ทุกการดำเนินการส่ง **session** ผ่านไป ซึ่งคือสิ่งที่ผูกมันเข้ากับ transaction ถ้าการอัปเดตใดล้มเหลว — สมมติว่าต้นทางมียอดคงเหลือไม่พอแล้วเรา throw — การโอนทั้งหมดจะถูก roll back และยอดคงเหลือทั้งสองไม่เปลี่ยนแปลง:

    session, err := client.StartSession()
    if err != nil {
        return err
    }
    defer session.EndSession(ctx)

    _, err = session.WithTransaction(ctx, func(sc mongo.SessionContext) (interface{}, error) {
        accounts := client.Database("library").Collection("accounts")
        if _, err := accounts.UpdateOne(sc,
            bson.M{"name": "Ada"}, bson.M{"$inc": bson.M{"balance": -3}}); err != nil {
            return nil, err
        }
        if _, err := accounts.UpdateOne(sc,
            bson.M{"name": "Linus"}, bson.M{"$inc": bson.M{"balance": 3}}); err != nil {
            return nil, err
        }
        return nil, nil
    })
    if err != nil {
        return err
    }
    ```

    let accounts = client
        .database("library")
        .collection::<Document>("accounts");

    session
        .start_transaction()
        .and_run((), |session, _| {
            let accounts = accounts.clone();
            async move {
                accounts
                    .update_one(doc! { "name": "Ada" }, doc! { "$inc": { "balance": -3 } })
                    .session(&mut *session)
                    .await?;
                accounts
                    .update_one(doc! { "name": "Linus" }, doc! { "$inc": { "balance": 3 } })
                    .session(&mut *session)
                    .await?;
                Ok(())
            }
            .boxed()
        })
        .await?;
    ```

ก่อนการโอน บัญชีทั้งสองมีสภาพดังนี้:

```json
[
  { "_id": 1, "name": "Ada", "balance": 10 },
  { "_id": 2, "name": "Linus", "balance": 4 }
]
```

หลังจาก commit สำเร็จ ค่าสามหน่วยได้ย้ายไปแล้วและยอดรวมยังคงเดิม ไม่มี reader คนใดเคยเห็นช่วงเวลาที่ค่าสามหน่วยของ Ada ออกไปจากเธอแล้วแต่ยังมาไม่ถึง Linus:

```json
[
  { "_id": 1, "name": "Ada", "balance": 7 },
  { "_id": 2, "name": "Linus", "balance": 7 }
]
```

**ใน Compass:** Compass ไม่ขับเคลื่อน transaction จาก document editor ของมัน แต่คุณสามารถเปิดแท็บ **mongosh** ที่ฝังอยู่ด้านล่างของหน้าต่าง แล้วรันสคริปต์ที่อิง session ตรงนั้นกับ connection เดียวกันได้

## เคล็ดลับและข้อควรระวัง

- **transaction ต้องใช้ replica set** `mongod` แบบ standalone ตัวเดียวรันมันไม่ได้ การตั้งค่า Docker บนเครื่องในคอร์สนี้รัน replica set แบบหนึ่ง node อยู่แล้ว เพื่อให้คุณลองตัวอย่างเหล่านี้ได้
- **ทำให้ transaction สั้น** transaction ถือ resource ไว้ตลอดอายุของมันและมีขีดจำกัดเวลาเป็นค่าเริ่มต้น ทำเฉพาะการเขียนที่ต้องเป็น atomic ภายในมัน และทำงานที่ช้า — การเรียกผ่านเครือข่าย การคำนวณหนัก — ไว้ภายนอก
- **มีต้นทุนด้านประสิทธิภาพจริง** transaction ช้ากว่าการเขียน single-document ที่เทียบเท่า และมันสามารถ abort ภายใต้ write conflict และต้องลองใหม่ ใช้มันที่ซึ่งความถูกต้องเรียกร้อง ไม่ใช่เป็นผ้าห่มปลอบใจ
- **ส่ง session ผ่านไปทุกที่** การดำเนินการภายใน callback ที่ลืมส่ง session จะรัน *นอก* transaction และจะไม่ถูก roll back

<Callout title="ให้ความสำคัญกับการออกแบบ single-document ก่อน">
ก่อนเขียน transaction ให้ถามว่าเอกสารสามารถปรับรูปร่างให้การเปลี่ยนแปลงพอดีกับเอกสารใดเอกสารหนึ่งได้หรือไม่ การเขียน single-document นั้นง่ายกว่า เร็วกว่า และเป็น atomic ฟรี ๆ transaction เป็นคำตอบเฉพาะเมื่อข้อมูลไม่สามารถนำมารวมกันได้จริง ๆ
</Callout>
