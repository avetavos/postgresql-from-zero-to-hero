การรับประกันที่แข็งแกร่งที่สุดที่ MongoDB ให้คุณก็เป็นการรับประกันที่เงียบที่สุดด้วย เพราะมันไม่มีต้นทุนและไม่ขออะไรจากคุณ ทุกการเขียนไปยัง **เอกสารหนึ่งฉบับ** เกิดขึ้นเป็นขั้นตอนเดียวที่แบ่งแยกไม่ได้ ถ้าการอัปเดตของคุณตั้งค่าสามฟิลด์ เพิ่มค่า counter และ push เข้า array ทั้งหมดในคราวเดียว ทุก reader จะเห็นการเปลี่ยนแปลงเหล่านั้นทั้งหมดหรือไม่เห็นเลย — ไม่มีทางเห็นสภาพที่นำไปใช้ครึ่งทาง และถ้ามีอะไรผิดพลาดระหว่างการเขียน เอกสารจะถูกทิ้งไว้ในสภาพเดิมเป๊ะอย่างที่เคยเป็นก่อนหน้า สิ่งนี้เป็นจริงไม่ว่าการเปลี่ยนแปลงจะลงลึกแค่ไหน: nested object และ array ภายในเอกสารเป็นส่วนหนึ่งของหน่วย atomic เดียวกัน

เราจะทำงานกับ `members` ของห้องสมุดเราต่อไป ลองนึกภาพเอกสารสมาชิกที่รวบรวมทุกอย่างเกี่ยวกับคนคนหนึ่งไว้ด้วยกัน — ค่าปรับ หนังสือที่กำลังยืมอยู่ และบันทึกกิจกรรมเล็ก ๆ:

```json
{
  "_id": "65f0b3d4e4b0a1c2e4b0a1d1",
  "name": "Ada",
  "fines": 4,
  "borrowed": ["Algorithms"],
  "activity": { "lastVisit": 2025, "visits": 12 }
}
```

## เขียนครั้งเดียว เปลี่ยนหลายอย่าง ก็ยังคง atomic

สมมติว่า Ada คืนหนังสือ จ่ายค่าปรับสองหน่วย และมาเยือนห้องสมุด — สามการเปลี่ยนแปลงเชิงตรรกะที่เกิดขึ้นในการมาเยือนครั้งเดียวนี้ เพราะทั้งหมดอยู่ใน **เอกสารเดียว** การอัปเดตครั้งเดียวจึงนำไปใช้พร้อมกัน ไม่มี transaction ให้เปิดและไม่มี session ให้จัดการ ตัว operator ทำงานเอง และการอัปเดตทั้งหมดลงในคราวเดียว:

ทุกส่วนของการเปลี่ยนแปลงลงพร้อมกัน และไม่มี reader คนใดเคยเห็นเอกสารในสภาพที่ ยกตัวอย่างเช่น ค่าปรับลดลงแล้วแต่หนังสือยังถูกยืมอยู่:

```json
{
  "_id": "65f0b3d4e4b0a1c2e4b0a1d1",
  "name": "Ada",
  "fines": 2,
  "borrowed": [],
  "activity": { "lastVisit": 2026, "visits": 13 }
}
```

ตัว update operator เองก็เป็น atomic `$inc` อ่านค่าปัจจุบันและบวกเข้าไปเป็นขั้นตอนเดียว ดังนั้นการเพิ่มค่าพร้อมกันสองครั้งจะไม่มีทางทำให้การอัปเดตหายไปจากการที่ทั้งคู่อ่านตัวเลขเริ่มต้นเดียวกัน คุณสมบัตินี้นี่เองคือเหตุผลว่าทำไม counter และยอดคงเหลือจึงควรอยู่ภายในเอกสารเดียว แทนที่จะคำนวณในแอปพลิเคชันของคุณแล้วเขียนกลับ

## ออกแบบโมเดลให้ข้อมูลที่เกี่ยวข้องกันอยู่ด้วยกัน

บทเรียนเชิงปฏิบัติคือเรื่องการออกแบบโมเดล เมื่อข้อมูลเปลี่ยนแปลงไปด้วยกัน ให้เก็บมันไว้ด้วยกัน แล้ว atomicity ฟรี ๆ จะครอบคลุมมัน เอกสารสมาชิกของเราทำเช่นนี้อยู่แล้ว: ค่าปรับ การยืม และบันทึกกิจกรรมล้วน embed อยู่ ดังนั้นสถานะทั้งหมดของสมาชิกหนึ่งคนจึงเคลื่อนไปเป็นหน่วยเดียว หากเราแยกบันทึกกิจกรรมออกไปเป็นคอลเลกชัน `activity` แยกต่างหากที่ key ด้วยสมาชิก การมาเยือนครั้งเดียวกันนั้นจะคร่อมสองเอกสารและบังคับให้เราต้องเลือกระหว่างการยอมเสีย atomicity หรือจ่ายค่า transaction

นี่คือหัวใจของเหตุผลที่ document model ของ MongoDB และ consistency model ของมันเข้ากันได้ นิสัยแบบ relational ที่ normalise ทุกอย่างลงในตารางแยกกันสร้างการอัปเดตหลายแถวที่ต้องใช้ transaction เพื่อให้สอดคล้องกัน ส่วนนิสัยแบบ document ที่ embed สิ่งที่เป็นของคู่กันเข้าด้วยกันเปลี่ยนหลาย ๆ กรณีเหล่านั้นให้เป็นการเขียน atomic ครั้งเดียว

## เคล็ดลับและข้อควรระวัง

- atomicity เป็นแบบต่อ **เอกสาร** ไม่ใช่ต่อ **การดำเนินการ** การ `updateMany` ครั้งเดียวที่แตะเอกสารหนึ่งพันฉบับคือการเขียน atomic ที่เป็นอิสระต่อกันหนึ่งพันครั้ง ไม่ใช่ขั้นตอน atomic ขั้นตอนเดียวขนาดใหญ่ — reader อีกคนสามารถเห็นบางส่วนที่นำไปใช้แล้วและบางส่วนที่ยังไม่ได้
- การอ่านไม่ block อยู่กับการเขียนเหล่านี้ reader จะได้เอกสารก่อนการเขียนหรือหลังการเขียน มันไม่มีทางรอสภาพที่เขียนครึ่งทาง เพราะไม่มีสภาพเขียนครึ่งทางให้รอ
- การ embed ไม่ได้ฟรีในทุกกรณี ถ้าเอกสารจะโตขึ้นโดยไม่มีขอบเขต — บันทึกกิจกรรมที่มีรายการนับล้าน — การแยกมันออกอาจเป็นทางเลือกที่ถูกต้อง แล้วคุณก็ยอมรับว่าการเปลี่ยนแปลงเหล่านั้นต้องใช้กลยุทธ์ความสอดคล้องที่ต่างออกไป

<Callout title="การ embed ให้ atomicity แก่คุณฟรี ๆ">
transaction ที่ถูกที่สุดคือ transaction ที่คุณไม่เคยต้องใช้ ออกแบบโมเดลข้อมูลที่เปลี่ยนแปลงไปด้วยกันให้อยู่ในเอกสารเดียว แล้วการอัปเดตธรรมดาครั้งเดียวจะรักษามันให้สอดคล้องกัน โดยไม่มี session ไม่มี commit และไม่มีต้นทุนเพิ่ม
</Callout>
